# Telegram AI Sales/Support Bot

Userbot на базе Pyrogram, который отвечает клиентам в Telegram от имени твоего аккаунта.
Использует Claude (Anthropic) для генерации ответов и RAG (Retrieval-Augmented Generation)
для поиска по базе знаний из твоих каналов и Google Drive.

---

## Что делает бот

- Отвечает клиентам в указанных группах/чатах автоматически
- Ищет релевантный контекст в базе знаний перед каждым ответом (RAG)
- Определяет язык клиента и отвечает на нём же
- Расшифровывает голосовые сообщения
- Читает документы (PDF, DOCX, XLSX)
- Извлекает контактные данные лидов и пересылает менеджеру
- Ведёт аналитику всех диалогов в PostgreSQL
- Каждый час обновляет базу знаний из Telegram каналов и Google Drive

## Переменные окружения (.env)

| Переменная | Где взять | Обязательно |
|---|---|---|
| `TELEGRAM_API_ID` | my.telegram.org → API development tools | Да |
| `TELEGRAM_API_HASH` | my.telegram.org → API development tools | Да |
| `TELEGRAM_PHONE` | твой номер в формате +79161234567 | Да |
| `ANTHROPIC_API_KEY` | console.anthropic.com → API Keys | Да |
| `DATABASE_URL` | строка подключения к PostgreSQL | Да |
| `QDRANT_URL` | http://localhost:6333 (локально) | Да |
| `ADMIN_USER_IDS` | @userinfobot в Telegram → твой ID | Да |
| `ALLOWED_CHAT_IDS` | ID групп/супергрупп где бот отвечает (через запятую) | Да |
| `MANAGER_USER_IDS` | Telegram ID живых менеджеров (через запятую) | Рекомендуется |
| `ALLOW_PRIVATE_CHATS` | true = бот отвечает в личке | Нет |
| `MANAGER_GROUP_ID` | ID группы для уведомлений о лидах и хэндоффах | Нет |
| `NOTIFICATION_CHAT_ID` | ID группы для зеркала всех ответов бота | Нет |
| `MONITORED_CHANNELS` | username каналов для RAG-базы знаний (через запятую) | Нет |
| `GOOGLE_DRIVE_FOLDER_ID` | ID папки в Google Drive | Нет |
| `GOOGLE_SERVICE_ACCOUNT_JSON` | путь к JSON сервисного аккаунта | Нет |
| `DRY_RUN` | true = бот не отвечает, только логирует | Нет |

### Как получить ID группы/канала
1. Запусти бота с `DRY_RUN=true` и пустым `ALLOWED_CHAT_IDS`
2. Напиши сообщение в нужной группе с другого аккаунта
3. В логах появится строка `[DRY_RUN] chat=-100XXXXXXXXX` — это и есть ID
4. Запиши в `ALLOWED_CHAT_IDS`

Для комментариев под постами канала: ID группы обсуждений, а не самого канала.

## Инструкция по запуску (с нуля)

### 1. Требования
- Python 3.11 (не 3.14 — несовместим с зависимостями)
- Docker Desktop (для Qdrant)
- PostgreSQL 18 с базой `bot_analytics`

### 2. Виртуальное окружение
```powershell
py -3.11 -m venv .venv
.venv\Scripts\activate
pip install -r requirements.txt
```

### 3. Скачать embedding модель (один раз, ~1 ГБ)
```powershell
python scripts/download_model.py
```

### 4. Запустить Qdrant
```powershell
docker run -d --name qdrant -p 6333:6333 qdrant/qdrant
```

Проверить: http://localhost:6333/healthz должен вернуть 200.

### 5. Заполнить .env
Скопировать `.env.example` → `.env`, заполнить все обязательные поля.

### 6. Запустить бота
```powershell
.venv\Scripts\python main.py
```

При первом запуске Telegram пришлёт код — вводишь в терминале.
Сессия сохраняется в файл `bot_session.session` — повторная авторизация не нужна.

### Что сделано в Шаге 1 (bot/client.py)
- Pyrogram userbot клиент с авторизацией через сессию
- Входящие сообщения: текст, голос, документы, аудио
- Фильтрация: только нужные типы чатов и контента
- Семафор `MAX_CONCURRENT_CHATS=25` — лимит параллельных диалогов
- Защита от дублей: `_processing` set
- Случайная задержка 2–5 сек (имитация человека)
- DRY_RUN режим: логирует, ничего не отправляет
- Зеркалирование ответов в `NOTIFICATION_CHAT_ID`
- Аналитика: открытие/закрытие диалогов по таймауту, запись сообщений
- **Протестировано:** бот видел реальное сообщение и залогировал DRY_RUN строку

### Что сделано в Шаге 1.5 (фильтры и передача диалога)

**Белый список чатов (`ALLOWED_CHAT_IDS`)**
- Личка: разрешена если `ALLOW_PRIVATE_CHATS=true`, белый список не применяется
- Группы/супергруппы: только chat_id из `ALLOWED_CHAT_IDS`
- Пустой список = отвечает везде (только для тестов, не для продакшена)

**Режимы ответа в группах (admin vs non-admin)**
- Бот-администратор → отвечает на все сообщения
- Бот не администратор → только на `@упоминание` или reply на его сообщение
- Статус проверяется через API, кэшируется на 1 час

**Передача диалога менеджеру (3 сценария)**

| Сценарий | Как детектируется |
|----------|------------------|
| Менеджер ответил **до** бота (во время задержки) | После `asyncio.sleep()` проверяем историю чата на новые сообщения от `MANAGER_USER_IDS` |
| Менеджер **вмешался после** бота | Следующее сообщение клиента — ответ на сообщение менеджера → `reply_to_message` |
| Оба сценария | `_handed_off[(chat_id, user_id)] = (имя, timestamp)` — бот молчит |

**Персистентность хэндоффов**
- Сохраняются в `data/handoffs.json` — выживают рестарт бота
- Запись старше 30 минут автоматически удаляется — бот возобновляет работу
- Диалог в PostgreSQL помечается статусом `handed_off`
- Уведомление в `MANAGER_GROUP_ID` или `NOTIFICATION_CHAT_ID`

## Важные предупреждения о рисках

### Блокировка аккаунта Telegram
- Userbot работает под ЛИЧНЫМ аккаунтом — при блокировке теряешь аккаунт
- **Рекомендация:** использовать отдельный номер телефона только для бота
- Задержки перед ответом 2–5 сек снижают риск
- Не делать массовую рассылку — только отвечать на входящие

### API ключи
- Никогда не публикуй `.env` в Git (файл уже в `.gitignore`)
- ANTHROPIC_API_KEY и TELEGRAM_API_HASH — как пароли
- При компрометации: немедленно сгенерировать новые

### Расходы на Claude API
- Лимит расходов: console.anthropic.com → Billing → Usage limits
- ~$2–3 в день при 100 активных клиентах
- Статистика: console.anthropic.com → Usage

## Шаг 7: Self-Service Sales Log (после Шага 6)

**Цель:** дать менеджерам инструмент регистрировать сделки прямо в боте, пока нет внешней CRM и единой платёжной системы.

Принимаемые способы оплаты: Payme, Click, Uzcard, Humo, банковские переводы, наличные.

### Команды для менеджеров (в группе менеджеров)

| Команда | Действие |
|---|---|
| `/sale @user product=LCD12-120 qty=1 price=1500000 payment=payme status=paid` | Сохраняет сделку → "Сделка #N сохранена" |
| `/support @user topic="доставка" status=solved` | Создаёт/закрывает тикет поддержки |
| `/stock LCD12-120 = 20` | Обновляет остаток товара на складе |
| `/report_today` | Краткий отчёт за день прямо в группу |
| `/export_excel` | Выгрузка всех событий в Excel-файл |

### Таблицы PostgreSQL (создать через Alembic в Шаге 3)

| Таблица | Назначение |
|---|---|
| `events` | Общий лог событий |
| `sales` | Продажи (продукт, цена, способ оплаты, статус) |
| `support_tickets` | Тикеты поддержки |
| `products` | Товары с ценой, себестоимостью и остатком |
| `managers` | Сотрудники |

## Режим DRY_RUN (тихий режим)

Устанавливается в `.env`:
```
DRY_RUN=true
```

Бот видит все сообщения и пишет в лог что бы ответил, но ничего не отправляет.
Используй при первом запуске и тестировании.

## Логи

Все события пишутся в терминал и в `logs/bot.log`.
Формат: `время | уровень | модуль | сообщение`

Уровни: INFO (штатная работа), WARNING (нештатно но не критично), ERROR (ошибка).
